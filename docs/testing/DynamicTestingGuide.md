# Dynamic Security Testing Guide

A practical guide to using the pentest agent for API security testing — from first setup through CI/CD automation.

## What This Is

The pentest agent is an AI-assisted security testing system built around a simple idea: **separate configuration from execution**.

- **Configuration** is interactive. You and the agent work together to capture traffic, define test flows ("use cases"), specify attacker profiles, and write authentication scripts. This happens at your own pace, incrementally.
- **Execution** is automated. Once configured, the agent runs every use case, produces machine-readable YAML results, and exits with a pass/fail code. This is the CI/CD path.

Everything lives in a self-contained `pentest/` folder that you can version-control, share with your team, and re-run across environments.

---

## Core Concepts

### The Pentest Folder

Every engagement lives in one folder. It contains everything needed to reproduce the test — no external state, no databases, no magic.

```
pentest/
  pentest-config.yaml       # Target URL, attacker profiles, global settings
  test-plan.yaml             # (Optional) Prioritised targets from threat model bridge

  har/                       # Traffic captures — one or more HAR files
    admin-flow.har
    user-checkout.har

  scripts/                   # Playwright scripts for login and token refresh
    login-standard.ts
    login-admin.ts

  usecases/                  # Test flows — one YAML per use case
    UC_LOGIN.yaml
    UC_CREATE_ORDER.yaml

  tokens/                    # Auth tokens (generated by scripts, gitignored)
    standard.json
    admin.json

  findings.yaml              # Results: vulnerabilities with proof
  coverage.yaml              # Results: what was tested, what wasn't

  diagrams/
    sequence.puml             # Visual model of the application flows
```

### Use Cases — The Core Unit

A **use case** is the fundamental building block. It represents one testable application flow.

Each use case contains three things:

1. **A baseline** — the legitimate "happy path" (a sequence of HTTP requests that should succeed)
2. **Attacks** — mutations of the baseline that should be rejected by the application
3. **Countermeasure validations** — specific security controls to verify (auth checks, rate limits)

For example, "Create Order" is a use case. Its baseline is: add item to cart, submit order, verify order. Its attacks include: access another user's order (IDOR), submit without authentication (auth bypass), inject extra fields (mass assignment), skip the cart step (business logic bypass).

Use cases have a lifecycle:

| Status | Meaning |
|--------|---------|
| `draft` | Skeleton created, needs tester input to fill in details |
| `configured` | Ready for automated execution |
| `disabled` | Temporarily excluded from runs |

Only `configured` use cases run in automated execution.

### Attacker Profiles

An attacker profile is a set of credentials representing a specific access level. You define them in `pentest-config.yaml`:

- **unauthenticated** — no credentials at all
- **standard_user** — a regular logged-in user
- **admin_user** — an administrator

Attacks work by taking a baseline step and replaying it with a *different* attacker profile. For instance: "this request succeeds as `standard_user` — does it also succeed as `unauthenticated`?" If yes, that's a finding.

### HAR Files

HAR (HTTP Archive) files are traffic captures from a browser session. They are the raw material the agent uses to discover endpoints and build use cases.

You can have **multiple** HAR files, each capturing a different flow or role. For example, one HAR from an admin session and another from a regular user session. Each is tagged with the attacker profile that generated it.

### Playwright Scripts

For authenticated testing, you need fresh tokens. Stale tokens from HAR files expire. Playwright scripts solve this by automating the login flow (including SSO, MFA, or custom auth) and outputting a JSON file with the current token.

These scripts are written once during configuration and run automatically before every test execution.

## Getting Started: Your First Pentest

### Step 1 — Create the Folder

Tell the agent:

> "Create a pentest folder at `./pentest/` for target `https://api.staging.example.com`"

The agent creates the folder structure and a starter `pentest-config.yaml` with the target URL. It will ask you for:

- The environment name (staging, production, local)
- At least one attacker profile — typically `unauthenticated` (needs no setup) and one authenticated profile

### Step 2 — Capture Traffic

You need at least one HAR file showing real application traffic. Two ways to get one:

**Option A — Browser DevTools:**
1. Open Chrome DevTools → Network tab
2. Navigate through the flow you want to test (e.g., login → create order → view order)
3. Right-click → "Save all as HAR with content"
4. Place the file in `pentest/har/`

**Option B — Playwright capture script:**
Tell the agent: *"Create a capture script for the checkout flow"*

The agent generates a Playwright script in `scripts/` that navigates the flow and saves a HAR file. You customise it for your app's specific UI, then run it.

Import the HAR:

> "Import HAR file `har/checkout-flow.har` — this is a standard user doing a checkout"

The agent parses it, discovers endpoints, and proposes grouping them into use cases.

### Step 3 — Set Up Authentication

For each authenticated attacker profile, you need a way to get fresh tokens.

**Option A — Static token (simplest):**
Set a token directly in config, referenced from an environment variable:

```yaml
admin_user:
  authenticated: true
  tokenSource: static
  token: "${ADMIN_TOKEN}"
```

**Option B — Playwright login script (recommended for CI/CD):**
Tell the agent: *"Create a login script for the standard user"*

The agent generates a template Playwright script. You fill in your app's login page selectors (email field, password field, submit button). The script outputs a token JSON file that the agent reads automatically.

Every auth script must produce a JSON file with this shape:

```json
{
  "token": "eyJ...",
  "expiresAt": "2026-03-26T15:30:00Z",
  "cookies": { "session_id": "abc123" }
}
```

### Step 4 — Build Use Cases

This is the core configuration step. You and the agent collaborate to define what gets tested.

**From a HAR file:**

> "Create use cases from `har/checkout-flow.har`"

The agent groups the HAR traffic into logical flows, proposes baseline steps, and suggests attacks. You review and refine.

**From a description:**

> "Create a use case for password reset: POST /auth/reset-request, then POST /auth/reset-confirm with the token from email"

The agent generates a draft with baseline steps and suggests relevant attacks (token reuse, IDOR on reset, rate limiting, brute force token guess).

**From a test plan (threat model integration):**

> "Import use cases from test-plan.yaml"

If a threat model bridge agent has generated a `test-plan.yaml`, the agent creates one draft use case per test target. These start as `draft` — you fill in the concrete request details.

### Step 5 — Review and Refine

For each draft use case, the agent walks you through:

1. **Baseline verification** — "Does this sequence of steps look right? Any missing headers or body fields?"
2. **Attack relevance** — "I suggest these 5 attacks. Any to remove? Any domain-specific ones to add?"
3. **Expected responses** — "What status code should a denied IDOR request return? 403? 404?"
4. **Data dependencies** — "Step 2 uses the `cartId` from Step 1's response. Is that the right field name?"
5. **Teardown** — "Should I delete the test order after each run? What endpoint and permissions?"

Once you're satisfied, set `status: configured`. That use case is now ready for automated execution.

### Step 6 — Run It

> "Execute all configured use cases"

The agent runs the full pipeline automatically — no further input needed.

## The Execution Pipeline

When you say "execute" (or trigger it from CI/CD), the agent runs these phases in strict order:

### Phase 0: Init and Validate

The agent reads `pentest-config.yaml` and all use case files. It checks that:

- Every attacker profile referenced by a use case exists in config
- Every profile that needs a script has one defined
- Every use case with `status: configured` has complete baseline steps

If anything is missing, it stops with **exit code 2** (configuration error) and reports exactly what's needed. No partial runs.

### Phase 1: Token Refresh

The agent runs every Playwright auth script to get fresh tokens. This ensures tests don't fail due to expired credentials.

```
Running scripts/login-standard.ts → tokens/standard.json ✓
Running scripts/login-admin.ts    → tokens/admin.json    ✓
```

If a login script fails (wrong password, UI changed, SSO down), the agent stops the use cases that depend on that profile and reports the failure.

### Phase 2: Baseline Verification

For each use case, the agent replays the baseline steps in order using the configured attacker profile's fresh tokens.

This step answers: **"Does the application's happy path still work?"**

- If all steps return the expected status codes and contain the expected response fields, the baseline passes.
- If any step fails, the agent skips that use case's attacks (you can't test attack mutations against a broken baseline) and reports it in coverage.

This is also when the agent discovers if the application has changed since you last configured the use case (e.g., a renamed endpoint, a changed response format).

### Phase 3: Attack Execution

For each use case with a passing baseline, the agent runs every attack:

**How attacks work:**

Each attack takes a baseline step and applies a **mutation** — changing the attacker profile, modifying the request path, merging extra fields into the body, or skipping prerequisite steps.

The attack defines what response *should* happen (e.g., `statusCode: 403`) and what response means a vulnerability was found (e.g., `failIf: 200`).

**Example: IDOR attack**

The baseline's Step 3 is `GET /api/v1/orders/{myOrderId}` as `standard_user`.
The IDOR attack replays it with a *different* order ID belonging to another user.
Expected: `403 Forbidden`. If the response is `200 OK` with the other user's data, that's a finding.

**Countermeasure validations** work similarly but test specific controls:

- "Is auth required?" — send the request without a token, expect `401`
- "Is rate limiting active?" — send 50 requests in 10 seconds, expect `429` after the threshold

**Nuclei scans** run `nuclei` (a template-based scanner) against the endpoints involved in the use case, checking for missing headers, CORS issues, known CVEs, and standard injection patterns.

### Phase 4: Teardown

For each use case that defines teardown steps, the agent runs cleanup requests — deleting test orders, reverting state changes, invalidating tokens. If cleanup fails, it's reported in coverage but doesn't affect the exit code.

### Phase 5: Reporting

The agent writes two YAML files:

**`findings.yaml`** — every vulnerability found, with full request/response proof:

```yaml
findings:
  - id: FINDING_001
    title: "IDOR — Accessed another user's order"
    severity: high
    usecase: UC_CREATE_ORDER
    attack: ATK_IDOR_ORDER
    proof:
      request: |
        GET /api/v1/orders/ORDER_OTHER_USER_123 HTTP/1.1
        Authorization: Bearer eyJ...
      response:
        statusCode: 200
        bodyExcerpt: |
          {"orderId": "ORDER_OTHER_USER_123", "userId": "other_user"}
    recommendation: "Enforce ownership checks on order retrieval."
```

**`coverage.yaml`** — a complete picture of what was tested:

```yaml
summary:
  totalUsecases: 5
  executed: 4
  disabled: 1
  totalAttacks: 18
  attacksPassed: 14
  attacksFailed: 3
  attacksSkipped: 1
  totalFindings: 3
  critical: 0
  high: 2
  medium: 1
```

**Exit codes:**

| Code | Meaning |
|------|---------|
| 0 | All attacks passed, no high/critical findings |
| 1 | High or critical findings found, or countermeasure validations failed |
| 2 | Configuration error — missing tokens, broken baseline, incomplete use cases |

## Use Case Anatomy — In Detail

Here's a complete use case with annotations explaining every section:

```yaml
# usecases/UC_CREATE_ORDER.yaml

id: UC_CREATE_ORDER                  # Unique identifier
title: "Create Order Flow"           # Human-readable name
description: |                       # What this flow does and tests
  User adds items to cart and submits an order.
  Tests authorization, input validation, and business logic.
status: configured                   # draft | configured | disabled

sourceHar:                           # Which HAR file(s) this was derived from
  - har/checkout-flow.har

# ──────────────────────────────────────────────
# BASELINE — the legitimate "happy path"
# ──────────────────────────────────────────────
baseline:
  profile: standard_user             # Run baseline as this attacker profile
  steps:
    - id: STEP_1
      description: "Add item to cart"
      request:
        method: POST
        path: "/api/v1/cart/items"
        headers:
          Content-Type: application/json
        body: |
          {"productId": "PROD_001", "quantity": 1}
      expectedResponse:
        statusCode: 200
        bodyContains: ["cartId"]     # Response must contain these strings

    - id: STEP_2
      description: "Submit order"
      request:
        method: POST
        path: "/api/v1/orders"
        headers:
          Content-Type: application/json
        body: |
          {"cartId": "${STEP_1.response.cartId}", "paymentMethod": "card"}
          # ^^^^^^ Data dependency: uses a value from Step 1's response
      expectedResponse:
        statusCode: 201
        bodyContains: ["orderId"]

    - id: STEP_3
      description: "Get order details"
      request:
        method: GET
        path: "/api/v1/orders/${STEP_2.response.orderId}"
      expectedResponse:
        statusCode: 200

# ──────────────────────────────────────────────
# ATTACKS — mutations of the baseline
# ──────────────────────────────────────────────
attacks:
  - id: ATK_IDOR_ORDER
    title: "IDOR — Access another user's order"
    category: "Elevation of Privilege"
    attackProfile: standard_user     # Same profile, different resource
    targetStep: STEP_3               # Mutate this step
    mutation:
      path: "/api/v1/orders/ORDER_OTHER_USER_123"
    expectedResult:
      statusCode: 403                # Should be denied
      failIf: 200                    # If we get 200, it's a vulnerability

  - id: ATK_UNAUTH_ORDER
    title: "Create order without authentication"
    category: "Elevation of Privilege"
    attackProfile: unauthenticated   # Different profile (no tokens)
    targetStep: STEP_2               # Same request, just no auth
    mutation: {}                     # No request changes — only the profile changes
    expectedResult:
      statusCode: 401
      failIf: 201

  - id: ATK_MASS_ASSIGN
    title: "Mass assignment — inject discount field"
    category: "Tampering"
    attackProfile: standard_user
    targetStep: STEP_2
    mutation:
      bodyMerge:                     # Merge extra fields into the original body
        discount: 100
        promoCode: "ADMIN_OVERRIDE"
    expectedResult:
      statusCode: 201
      bodyMustNotContain: ["discount", "ADMIN_OVERRIDE"]

  - id: ATK_SKIP_CART
    title: "Business logic bypass — order without cart"
    category: "Tampering"
    attackProfile: standard_user
    skipSteps: [STEP_1]              # Don't run Step 1 first
    targetStep: STEP_2
    mutation:
      body: |
        {"cartId": "NONEXISTENT", "paymentMethod": "card"}
    expectedResult:
      statusCode: 400
      failIf: 201

  - id: ATK_NUCLEI
    title: "Automated scan on order endpoints"
    category: "Automated"
    type: nuclei                     # Runs nuclei scanner instead of curl
    targetEndpoints:
      - "/api/v1/cart/items"
      - "/api/v1/orders"

# ──────────────────────────────────────────────
# COUNTERMEASURE VALIDATIONS — specific controls to test
# ──────────────────────────────────────────────
countermeasureValidations:
  - id: CM_ORDER_AUTH
    title: "Order endpoint requires authentication"
    test: "Send POST /api/v1/orders without auth header"
    expectedStatusCode: 401
    attackProfile: unauthenticated
    targetStep: STEP_2

  - id: CM_RATE_LIMIT
    title: "Order creation is rate-limited"
    test: "Send 50 POST /api/v1/orders in 10 seconds"
    expectedStatusCode: 429
    attackProfile: standard_user
    targetStep: STEP_2
    burstCount: 50
    burstWindowSeconds: 10

# ──────────────────────────────────────────────
# TEARDOWN — cleanup after the use case runs
# ──────────────────────────────────────────────
teardown:
  - description: "Delete test order"
    request:
      method: DELETE
      path: "/api/v1/orders/${STEP_2.response.orderId}"
    profile: admin_user              # May need elevated permissions
```

## Attack Types Reference

| Attack Pattern | What It Tests | How It Works |
|----------------|---------------|--------------|
| **IDOR** | Broken access control | Change resource IDs to access another user's data |
| **Vertical privilege escalation** | Auth enforcement | Use a low-privilege (or no) token on a restricted endpoint |
| **Horizontal privilege escalation** | Resource isolation | Use User A's token to access User B's resources |
| **Mass assignment** | Input validation | Add unexpected fields (role, isAdmin, discount) to POST/PUT requests |
| **Business logic bypass** | Flow integrity | Skip required steps in a multi-step process |
| **Auth bypass** | Authentication | Remove or corrupt auth headers, use expired/forged tokens |
| **Rate limit check** | DoS protection | Send a burst of requests and verify throttling kicks in |
| **Nuclei scan** | Common vulnerabilities | Automated check for missing headers, CORS, CVEs, injection patterns |

## Threat Model Integration

The pentest agent works standalone, but it can also integrate with a threat model via the **bridge agent** (`pentest-tm-bridge`). This creates a two-way data flow:

```
Threat Model (.yaml)                      Pentest Folder
       │                                        │
       │  ── bridge generates ──→  test-plan.yaml
       │                                        │
       │                              (tester configures use cases)
       │                                        │
       │                              (agent executes, produces results)
       │                                        │
       │  ←── bridge reads ────  findings.yaml
       │                          coverage.yaml
       │
       ▼
  tm-updates.yaml  (proposed changes to the threat model)
```

**Direction 1 — Threat model drives the pentest:**
The bridge reads the threat model, extracts threats marked `pentestTestable: true`, and generates a `test-plan.yaml`. The pentest agent converts these into draft use cases for you to configure.

**Direction 2 — Pentest results update the threat model:**
After execution, the bridge reads `findings.yaml` and `coverage.yaml` and proposes updates: new threats for vulnerabilities not in the model, countermeasure status changes, broken assumptions.

The two folders can be co-located (pentest as a subfolder of the threat model directory) or completely separate.

## The Improvement Loop

The pentest folder is designed to get better over time:

```
┌──────────────────────────────────────────────────┐
│                Improvement Loop                   │
│                                                  │
│  CONFIGURE ──→  usecases/*.yaml  ──→  EXECUTE    │
│      ↑                                    │      │
│      │                                    ▼      │
│      └──── coverage gaps  ←────  coverage.yaml   │
│            new HAR files          findings.yaml   │
│            tester feedback                       │
└──────────────────────────────────────────────────┘
```

After each run:

- **coverage.yaml** shows unmapped endpoints (traffic the agent saw but no use case covers), skipped tests, and failed baselines
- **findings.yaml** reveals vulnerabilities that may need new use cases to track regression
- **New HAR files** from updated application flows feed new use cases
- **Tester feedback** refines attack parameters, adds domain-specific mutations

Each CONFIGURE session makes the next EXECUTE run more comprehensive.

## Querying Results with CLI Tools

Since findings and coverage are YAML, you can query them with standard CLI tools:

```bash
# Count findings by severity
yq '.findings[].severity' findings.yaml | sort | uniq -c

# List all failed attacks
yq '.usecaseCoverage[].attacks[] | select(.result == "fail")' coverage.yaml

# Get all critical/high findings
yq '.findings[] | select(.severity == "critical" or .severity == "high")' findings.yaml

# List unmapped endpoints (discovered but not tested)
yq '.endpointCoverage.unmapped[]' coverage.yaml

# Check which countermeasures failed
yq '.usecaseCoverage[].countermeasures[] | select(.result == "fail")' coverage.yaml

# Summary stats
yq '.summary' coverage.yaml
```

## Safety Rules

The agent follows strict safety policies:

1. **Never test production** without your explicit confirmation
2. **Never store real credentials** in YAML — always use `${ENV_VAR}` references
3. **Never fabricate findings** — every vulnerability has request/response proof
4. **Always run teardown** to clean up test data
5. **Respect disabled use cases and skip overrides** — they exist for a reason
6. **Stop and ask** if unexpected admin access or out-of-scope data is encountered
7. **Never run DoS/brute-force** unless explicitly approved on an isolated environment
8. **In EXECUTE mode, never prompt** — if config is incomplete, fail with exit code 2
